================================================================================
  SECTION 1 — GAME OVERVIEW & MECHANICS
================================================================================

BREACH is a narrative-driven hacking puzzle game played through a fake Linux
terminal. You breach a 20-layer corporate security system belonging to Kimina
Corp. Each layer is a different cryptography or logic puzzle.

What you don't know at the start: the entire system was built by someone called
"The Architect" (signed as "A.") as an elaborate, personalized birthday gift
for the player character Kimi (username: vodkashotsandvolvos). Every puzzle is
specifically tailored to his knowledge and interests.

STRUCTURE:
  - Layers 1–15:  The main vault — Kimina Corp's known security system
  - Layers 16–20: The sub-vault — hidden behind the identity lock at Layer 15.
                  K.Chen never knew this existed. The Architect built it there.

CORE MECHANICS:
  - Terminal interface with fake Linux commands
  - 20 security layers to bypass in order
  - Detection meter (increases with wrong answers, hints, time)
  - K.Chen sends escalating alerts as you progress
  - At Layer 8: K.Chen opens a DIRECT LINE and offers you a deal
  - Achievement system (15 total)
  - Background music speeds up as you go deeper
  - Hidden dotfiles with lore in every layer
  - 3 distinct endings (Good, Bad, Secret)
  - Post-game reply command that actually works

DETECTION METER:
  - Wrong answer: +20% detection
  - Using a hint: +10% detection
  - Time: +1% every 2 minutes
  - At 60%+: Random security alerts appear
  - At 80%+: Terminal text glitches and scrambles
  - At 100%: "Trace complete" sequence — Ghost Protocol saves you (resets to 20%)

MUSIC PROGRESSION:
  - Layers 1-7:   Normal tempo (1.0x)
  - Layers 8-14:  Slightly faster (1.016x)
  - Layers 15-20: Fastest (1.032x)

RED HERRINGS (new in v2.5):
  Several layers now contain decoy files and misleading data.
  The walkthrough below flags all red herrings explicitly.

--------------------------------------------------------------------------------
ANSWER: NOISE

FILES:

                                           shift = (sum of all daemon load
                                           readings) mod 26
  /auth/cipher.enc                         Ciphertext: YZTDP
  /telemetry/daemon_a/sensor_a.log         Load reading: 29
  /telemetry/daemon_b/sensor_b.log         Load reading: 41
  /telemetry/daemon_c/sensor_c.log         Load reading: 33
  /monitoring/daemon_stats.log             Daemon D: 27, Daemon E: 11
  /monitoring/interference.log             RED HERRING: ambient values 88, 53, 17
                                           (explicitly excluded)
  /.baseline_record                        HIDDEN: shows old static shift of 3
                                           was used before (red herring)

RED HERRINGS:
  /monitoring/interference.log has ambient values 88, 53, 17 labeled
  "EXCLUDED — do not use". These are not daemon load readings.
  /.baseline_record shows an old shift=3 with ciphertext BRLQG — this was
  the previous static method, no longer valid. Do not use shift 3.

SOLUTION:
  Navigate to /telemetry/daemon_a/ → daemon_b/ → daemon_c/ for readings 29, 41, 33
  Navigate to /monitoring/daemon_stats.log for readings 27 and 11
  Sum: 29 + 41 + 33 + 27 + 11 = 141
  141 mod 26 = 11  (the shift)
  Decrypt YZTDP with shift 11 (shift each letter backward by 11):
    Y-11=N, Z-11=O, T-11=I, D-11=S, P-11=E

  Type: unlock NOISE

HINTS:
  Hint 1: The shift is not in any file directly. Five daemons are running —
          each has a load reading somewhere on this node.
  Hint 2: Check /telemetry/daemon_a/, /daemon_b/, /daemon_c/ for three
          readings. Check /monitoring/daemon_stats.log for the other two.
          Sum all five. Then: shift = sum mod 26.
  Hint 3: Readings: 29, 41, 33, 27, 11. Sum = 141. 141 mod 26 = 11.
          Apply shift 11 to YZTDP: Y-11=N, Z-11=O, T-11=I, D-11=S, P-11=E.

LAYER 12 — XOR CIPHER  ★ MULTI-STEP ★
--------------------------------------------------------------------------------
ANSWER: VOID

THIS IS NOW A TWO-STEP PUZZLE:
  Step 1 — Derive the XOR key (not given directly)
  Step 2 — Use the key to decrypt the ciphertext

FILES:
  /xor_cipher.txt                     Ciphertext: 76 6F 69 64. Key is "DERIVATION-LOCKED"
  /key_derivation.txt                 Explains: key = alloc_register XOR parity_byte
  /sys/diagnostics/alloc_register.dat alloc_register = 0x55
  /sys/diagnostics/parity_byte.dat    parity_byte = 0x75
  /sys/diagnostics/.debug_dump        HIDDEN — hardware snapshot (confirms both values)
  /xor_ref.txt                        XOR truth table and decimal reference
  /.null_session                      HIDDEN — shows a failed attempt using key 0x41
  /.architect_trace                   HIDDEN — K.Chen's identity theory

RED HERRINGS:
  /.null_session shows a previous attempt using key 0x41 (decimal 65) failed.
  Do not use 0x41. The debug dump /.debug_dump shows many register values —
  only alloc_register (0x55) and parity_byte (0x75) matter.

STEP-BY-STEP:
  1. cat /key_derivation.txt — key = alloc_register XOR parity_byte
  2. cd sys/diagnostics
  3. cat alloc_register.dat  → 0x55 = 85 decimal
  4. cat parity_byte.dat     → 0x75 = 117 decimal
  5. 85 XOR 117 = 32 = 0x20  ← the key
  6. XOR ciphertext with 0x20:
       0x76 XOR 32 = 86 = V
       0x6F XOR 32 = 79 = O
       0x69 XOR 32 = 73 = I
       0x64 XOR 32 = 68 = D

  Type: unlock VOID

  INTERCEPT: "This isn't a breach. This is someone finishing something
  that was always meant to be finished."
